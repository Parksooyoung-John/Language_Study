========================================
3. 프로젝트가 시작된 이유
========================================

출발점은 거창한 서비스 개발이 아니라 개인적인 학습 불편이었다.

중국어 말하기를 해야 한다는 생각은 있지만, 실제로 시작하려면 여러 마찰이 있었다.

- 오늘 무엇을 공부할지 정해야 한다.
- HSKK 형식에 맞는 문장과 질문을 찾아야 한다.
- 병음과 뜻을 같이 정리해야 한다.
- 녹음하고 다시 들어보는 흐름이 귀찮다.
- 내 답변이 어느 부분에서 부족한지 바로 알기 어렵다.
- 학습 기록과 복습 포인트를 계속 관리해야 한다.

그래서 목표를 단순하게 잡았다.

내가 원할 때 모바일에서 바로 열고, 오늘 해야 할 HSKK 말하기 연습만 시작하게 만들자.

이후 프로젝트는 다음 방향으로 발전했다.

1. Codex/Harness가 매일 HSKK 레슨을 생성한다.
2. 생성된 레슨을 GitHub Pages 모바일 페이지로 배포한다.
3. 사용자는 홈 화면 바로가기로 앱처럼 접속한다.
4. 레슨을 본 뒤 바로 AI Speaking Trainer로 이동한다.
5. Trainer에서 녹음, 내 녹음 재생, STT, AI 평가, 원어민 발음 비교, 교정 답변 발음 비교까지 수행한다.
6. 결과는 History와 JSON export로 남길 수 있다.
7. JSON export는 향후 Harness import와 복습 루프 자동화를 염두에 둔 구조다.

========================================
5. 사용자가 경험하는 실제 흐름
========================================

사용자 경험은 다음 순서다.

1. 스마트폰 홈 화면의 바로가기 아이콘을 누른다.
2. GitHub Pages 모바일 학습 페이지가 열린다.
3. 오늘의 HSKK 레슨을 확인한다.
4. 한자, 병음, 한국어 뜻, 그림 묘사, 질문 답변을 훑어본다.
5. 필요하면 자기채점/메모 영역에 학습 포인트를 남긴다.
6. 레슨 하단의 “녹음하고 AI 평가 받기”를 누른다.
7. HSKK AI Trainer로 이동한다.
8. Trainer에서 오늘 레슨 내용을 보면서 녹음한다.
9. 녹음 완료 후 내 녹음을 직접 들어본다.
10. 마음에 들지 않으면 다시 녹음한다.
11. 평가하기를 누르면 STT가 실행된다.
12. STT transcript가 생성된다.
13. transcript를 기반으로 원어민 스타일 TTS 발음을 들을 수 있다.
14. AI 평가가 실행된다.
15. AI 코칭 리포트에서 예상 점수, 강점, 약점, 교정 답변, 다음 연습 추천을 확인한다.
16. 교정 답변도 TTS로 들어보고 내 답변과 비교한다.
17. 예상문제 Test에서는 현재 레슨 주제 기반 질문을 보고 다시 말할 수 있다.
18. 결과를 JSON으로 export하거나 History에 저장한다.
19. 레슨 계속 보기로 돌아가 학습 루프를 이어간다.

6. Main 장표 3장 구성 제안
========================================

Main Slide 1: Hook

제목:
공부 결심보다 쉬운 중국어 말하기

부제:
스마트폰에서 열고, 말하고, 바로 피드백 받는 HSKK AI 루틴

시각 구성:
- 실제 iPhone 홈 화면 또는 GitHub Pages 바로가기 화면을 크게 사용
- HSKK Daily, 오늘의 HSKK 30분, AI Trainer, 녹음 버튼 같은 실제 화면 요소 강조
- 기술 구조보다 사용자가 느끼는 경험을 먼저 보여준다

핵심 키워드:
- 바로 시작
- 모바일 학습
- 말하기 연습
- AI 피드백
- 원어민 발음 비교

발표 방향:
이 프로젝트는 “더 열심히 공부하자”보다 “공부 시작 전 마찰을 줄이자”에서 시작했다. 스마트폰 홈 화면에서 바로 열고, 오늘 레슨을 보고, 녹음해서 AI 피드백을 받는 흐름을 만들었다.

Main Slide 2: From Tap To Coaching

슬라이드 메시지:
한 번의 탭이 오늘의 레슨, 녹음, STT, AI 코칭, 발음 비교로 이어진다.

시각 구성:
왼쪽에서 오른쪽으로 5단계 흐름.

1. Tap: 홈 화면 바로가기
2. Learn: GitHub Pages 오늘의 레슨
3. Speak: 브라우저 녹음과 내 녹음 재생
4. Coach: STT + AI 평가 리포트
5. Compare: 원어민 발음과 교정 답변 발음 비교

핵심 키워드:
- Lesson
- Record
- STT
- AI Feedback
- Pronunciation Compare

발표 방향:
사용자는 GitHub Pages에서 오늘 레슨을 보고, 레슨 하단의 버튼으로 Trainer에 들어간다. 같은 레슨 내용을 보면서 녹음하고, 내 녹음을 들어본 다음 STT와 AI 평가를 실행한다. 결과에서는 예상 점수와 코칭 리포트, 교정 답변, 모범 발음을 확인한다.

Main Slide 3: Harness가 하는 일

슬라이드 메시지:
단순 생성이 아니라, 매일 반복되는 학습 운영 체계.

시각 구성:
- 중앙: 나의 HSKK 학습 루틴 또는 모바일 Trainer 화면
- 주변: Agent 또는 작업 단계
- 원형 Loop 구조

Harness Loop:
1. Plan: HSKK 형식과 이전 학습 상태 확인
2. Generate: 오늘의 레슨 생성
3. Publish: GitHub Pages 모바일 페이지 갱신
4. Practice: 모바일에서 학습, 녹음, AI 평가
5. Export: 세션 JSON 저장 또는 export
6. Review: 약점과 복습 큐에 반영

핵심 키워드:
- Multi-Agent
- Daily Loop
- Mobile-first
- AI Coaching
- JSON Export
- Review Queue

발표 방향:
핵심은 AI가 한 번 답을 만들어 주는 것이 아니다. Harness는 매일 반복되는 학습 운영을 담당한다. 레슨을 만들고, 모바일 페이지를 갱신하고, 학습 결과를 다음 개선으로 연결할 수 있는 구조를 만든다.

========================================
7. Harness 구조 설명 옵션
========================================

Option 1: Learning Factory

비유:
학습 목표와 피드백이라는 재료가 들어오면, Agent들이 각 작업대에서 오늘의 학습 콘텐츠와 코칭 루틴을 만들어 모바일로 포장해 준다.

구성:
- 왼쪽: HSKK 목표, 시험 형식, 이전 레슨, 자기채점 메모, 녹음/평가 JSON
- 중앙: Planner, Lesson Tutor, Trainer Coach, Review Coach, Publisher
- 오른쪽: 모바일 레슨, AI Speaking Trainer, 예상문제 Test, JSON export, 복습 계획

발표용 설명:
Harness를 공장처럼 설명하면 비개발자도 이해하기 쉽다. 학습 목표와 이전 결과가 재료로 들어가고, 여러 Agent가 역할을 나눠 레슨, 테스트, 코칭, 배포 결과물을 만든다.

Option 2: Agent Workers

비유:
작은 AI 작업반이 매일 아침 오늘의 말하기 연습 세트를 준비하고, 나는 스마트폰에서 바로 연습한다.

Agent 역할:
- 커리큘럼 일꾼: 학습 목표와 시험 구조 확인
- 레슨 일꾼: 한자, 병음, 뜻, 질문 준비
- 평가 일꾼: STT와 AI 코칭 연결
- 발음 일꾼: 원어민 스타일 TTS와 교정 답변 발음 준비
- 복습 일꾼: 약점과 다음 연습 추천 정리

발표용 설명:
Agent를 “일꾼”으로 표현하면 복잡한 멀티에이전트 개념을 쉽게 풀 수 있다. 사용자는 작업 과정 전체를 신경 쓰지 않고, 완성된 오늘의 연습 세트만 모바일에서 열면 된다.

Option 3: Learning Lunchbox

비유:
여러 Agent가 학습 재료를 손질해서 오늘 바로 먹을 수 있는 30분 HSKK 학습 도시락으로 포장한다.

구성:
- 재료: 목표, 시험 형식, 이전 레슨, 녹음 결과, 약점
- 요리 과정: 커리큘럼 레시피, 문장 카드, 예상문제, 발음 비교, 복습 추천
- 완성품: Part 1 반복, Part 2 그림묘사, Part 3 질문 답변, 녹음/AI 평가, 원어민 발음 비교

발표용 설명:
학습 도시락 비유는 “매일 바로 실행 가능한 30분 학습 세트”라는 메시지를 전달하기 좋다.

Option 4: Loop Engine

비유:
학습은 끝나는 이벤트가 아니라 다음 레슨의 입력으로 돌아가는 루프다.

Loop:
- Generate Lesson
- Practice on Mobile
- Record Voice
- AI Feedback
- Export JSON
- Review Queue
- Next Lesson

발표용 설명:
이 프로젝트의 장점은 한 번 생성하고 끝나는 것이 아니라 매일 반복 가능한 운영 루프를 만든다는 점이다.

주의:
현재 모바일 입력값이 완전 자동으로 Codex/Harness에 들어가지는 않는다. 발표에서는 “JSON export와 Harness import를 염두에 둔 구조”라고 표현해야 한다.

9. STT/AI 평가/TTS 구조

녹음:

AI 평가:

- AI가 rubric 기반 JSON을 반환한다.
- rubric:
  - task_completion
  - fluency
  - grammar
  - vocabulary
  - pronunciation
  - overall comments
- 결과:
  - estimated_score
  - strengths
  - weaknesses
  - corrected_answer
  - review_recommendations

TTS:

- 기본 모델: gpt-4o-mini-tts
- 기본 voice: coral
- transcript 기반 “원어민 발음 듣기”
- corrected_answer 기반 “교정 답변 발음 듣기”

중요 표현:
원어민 발음은 실제 사람 원어민 녹음이 아니라 OpenAI TTS 기반 모범 발음이다.

11. 레슨 페이지와 Trainer 연결
========================================

레슨 페이지:

Trainer:

- 오늘 레슨 모드에서는 lesson JSON을 읽어 오늘 말할 내용을 보여준다.
- 예상문제 Test 모드에서도 lesson JSON이 있으면 해당 레슨 기반 문제를 보여준다.

사용자에게 중요한 가치:
레슨을 따로 보고, 평가 페이지에서 다시 문제를 찾을 필요가 없다. 방금 본 레슨 주제가 그대로 Trainer에 연결된다.

12. 자기채점/피드백 메모의 현재 상태

사용 목적:

- 예: “성조가 흔들림. 다음에는 이유를 한 문장 더 붙이기.”
- 발표영상에서는 학습 루프의 시작점으로 보여준다.

- 향후 자동 반영을 하려면 서버 저장소, GitHub import, 또는 Harness import 자동화가 필요하다.

발표용 표현:

18. 예상 질문과 답변
========================================

Q1. GitHub Pages를 모바일 홈 화면 바로가기로 쓰고 있는데, 범용 앱처럼 쓰려면 어떻게 해야 하나?

답변:
현재는 개인용 모바일 웹앱에 가깝다. 범용 앱처럼 확장하려면 PWA 구조를 추가하는 것이 다음 단계다. 필요한 요소는 web app manifest, 앱 아이콘, service worker, 오프라인 캐시, 사용자별 진행 상태 저장, 피드백 제출 저장소 또는 데이터베이스, 배포/권한/보안 정책이다.

짧은 답변:
지금은 개인용 모바일 웹앱이고, 범용 앱처럼 확장하려면 PWA와 사용자별 데이터 저장 구조를 추가하면 된다.

Q2. 자기채점과 피드백을 입력하면 다음 학습에 자동 반영되나?

답변:
현재 완전 자동 반영은 아니다. 모바일 페이지와 Trainer는 학습, 녹음, 평가, JSON export까지 지원한다. 하지만 사용자가 입력한 자기채점 메모나 학습 피드백이 자동으로 Codex/Harness에 들어가는 구조는 아직 구현되어 있지 않다. 현재는 JSON export 또는 사용자가 Codex에 피드백을 전달하는 방식이 안정적이다.

짧은 답변:
현재는 반자동 루프다. 학습 결과를 JSON이나 Codex 입력으로 넘기면 다음 레슨과 복습 계획에 반영할 수 있고, 완전 자동 반영은 향후 서버 저장소나 GitHub import 자동화가 필요하다.

Q3. NotebookLM이나 자료작성 Tool과 Codex가 바로 연계 가능한가?

답변:
현재 프로젝트 기준으로는 직접 자동 연동이 아니라 파일 기반 연계가 가장 안정적이다. Codex가 NotebookLM용 Markdown 소스를 작성하고, 사용자가 NotebookLM에 업로드하면 NotebookLM이 발표자료, 요약, Q&A를 생성한다. 필요하면 Codex가 결과 PPT를 검토하고 수정 소스를 다시 만들 수 있다.

짧은 답변:
지금은 Codex가 소스를 만들고 NotebookLM이 그 소스를 읽는 방식이 가장 안정적이다. 직접 연동은 별도 커넥터나 API가 필요하다.

Q4. STT와 TTS를 쓰면 비용이 계속 발생하나?

답변:
녹음 자체와 내 녹음 재생은 로컬 처리라 비용이 없다. 녹음 파일을 STT로 변환하거나, transcript와 교정 답변을 TTS로 만들 때 API 호출이 발생한다. 그래서 사용자가 평가하기 버튼을 눌렀을 때만 호출하도록 설계했다.

짧은 답변:
녹음 자체는 로컬이고, STT/AI 평가/TTS를 실행할 때만 API가 호출된다.

Q5. Cloudflare Worker를 쓰지 않고 Vercel API를 쓰는 이유는?

답변:
초기에는 Cloudflare Worker를 사용했지만, OpenAI STT 호출에서 지역 제한 오류가 발생했다. Smart Placement도 시도했으나 해결되지 않아, OpenAI API 호출 위치를 안정적으로 고정할 수 있는 Vercel Serverless Function으로 이전했다.

짧은 답변:
Cloudflare Worker에서 OpenAI STT 지역 제한 오류가 발생해, 운영 백엔드는 Vercel API로 전환했다. GitHub Pages는 화면을 담당하고, Vercel은 OpenAI API 호출을 담당한다.

Q6. HSKK 예상문제 Test는 상용 문제은행을 쓰나?

답변:
아니다. 저작권이 있는 상용 문제은행을 쓰지 않는다. 기본 예상문제와 현재 레슨 JSON의 질문, 그림묘사, 반복 문장을 재조합해 HSKK 스타일 연습 문제로 제공한다.

짧은 답변:
상용 문제은행이 아니라, 자체 생성 레슨과 HSKK 형식 기반의 연습 문제다.

Q7. 원어민 발음 비교는 정확히 무엇인가?

답변:
사용자의 녹음이 STT로 transcript가 되면, 그 transcript를 OpenAI TTS로 다시 생성해 모범 발음처럼 들려준다. 평가 후에는 AI가 다듬은 교정 답변도 TTS로 들을 수 있다.

비교 대상:
- 내 녹음
- STT transcript 기반 모범 발음
- AI 교정 답변 기반 모범 발음

짧은 답변:

19. 발표스크립트 기본 내러티브
========================================

Opening:
언어학습에서 가장 어려운 것은 공부 자체보다 시작까지의 마찰이다. 오늘 무엇을 할지 찾고, 자료를 고르고, 기록을 남기고, 복습을 챙기는 과정을 반복하다 보면 말하기 연습은 뒤로 밀린다.

Problem:
HSKK 말하기를 준비하려면 매일 조금씩 말해야 하는데, 그 루틴을 혼자 유지하기가 어렵다. 특히 모바일에서 바로 열고, 바로 말하고, 바로 피드백 받는 구조가 필요했다.

Solution:
그래서 Codex와 Harness로 매일 HSKK 레슨을 만들고, GitHub Pages 모바일 페이지로 배포하고, AI Speaking Trainer에서 녹음과 평가까지 연결했다.

Demo:
사용자는 홈 화면 바로가기를 누르고 오늘 레슨을 본다. 레슨 하단에서 Trainer로 이동해 녹음하고, 내 녹음을 들어본 뒤 STT와 AI 평가를 실행한다. 결과 화면에서는 강점, 약점, 교정 답변을 보고, 원어민 스타일 발음과 교정 답변 발음을 비교한다.

Harness:
뒤에서는 Harness가 매일 반복되는 학습 운영을 담당한다. 레슨 생성, 모바일 페이지 갱신, 복습 큐, JSON export를 연결해 학습 결과가 다음 개선으로 돌아갈 수 있게 만든다.

Closing:
핵심은 AI가 답을 한 번 주는 것이 아니다. AI가 매일 학습 준비와 피드백 구조를 만들고, 사람은 실제 말하기 연습에 집중하도록 만드는 것이다.
